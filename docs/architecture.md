# 기술 아키텍처 요약

## 1. 모노레포 구조

```text
longf/
├─ apps/
│  ├─ api/        FastAPI, SQLAlchemy, Celery, Redis, PostgreSQL
│  └─ web/        Next.js, TypeScript, Tailwind, React Query, Zustand
├─ docs/          아키텍처/스키마 문서
├─ docker-compose.yml
├─ .env.example
└─ README.md
```

## 2. 계층 분리 원칙

### Domain
- 순수 규칙과 계약만 둡니다.
- 이슈 랭킹, 수치 근거 검증 같은 핵심 비즈니스 규칙은 외부 라이브러리 의존 없이 유지합니다.
- 제공자 인터페이스는 Protocol 기반 포트로 정의하여 교체 가능성을 보장합니다.

### Application
- 유스케이스 조합, 요청/응답 스키마, 서비스 오케스트레이션을 담당합니다.
- 입력 검증과 출력 구조 표준화를 담당합니다.
- 프레젠테이션 계층은 오직 application service를 호출합니다.

### Infrastructure
- SQLAlchemy 모델, Celery 작업, 로컬/S3 저장소, 외부 API 어댑터를 담당합니다.
- Claude, ECOS, KOSIS, FRED, OECD, Yahoo, Investing, Seeking Alpha, 이미지 생성기, Veo 워크플로는 모두 어댑터로 분리했습니다.
- 실제 연동이 불확실한 부분은 mock adapter와 TODO 계약으로 표시했습니다.

### Presentation
- FastAPI REST 라우터와 Next.js UI가 각각 프레젠테이션 계층 역할을 맡습니다.
- 프런트는 한국어 전용 대시보드와 편집형 워크플로 화면을 제공합니다.

## 3. 핵심 설계 포인트

### 프로젝트 중심 영속성
- 프로젝트가 상위 집합이며 이슈, 통계, 시장 데이터, 스냅샷, 대본, 장면, 이미지, 영상, 작업 로그가 모두 프로젝트에 귀속됩니다.
- 설정, 프롬프트 스냅샷, 근거, 생성 자산, 버전 이력을 프로젝트 문맥에서 복원할 수 있게 설계했습니다.

### 근거 우선 스크립트 생성
- 스크립트 생성기는 선택한 이슈와 공식 통계 근거를 우선 사용합니다.
- 숫자 주장 검증은 도메인 서비스에서 독립 수행되며, 발표일과 신선도 기준을 함께 확인합니다.
- 시장 포털 자료는 보조 자료로 구분해 메타데이터를 유지합니다.

### 교체 가능한 외부 제공자
- 각 제공자는 공통 포트 뒤에 숨어 있으며 코어 도메인은 제공자 세부사항을 모릅니다.
- 향후 실제 연동 시에도 application service는 어댑터 교체만으로 유지됩니다.

### 비동기 작업 처리
- 이슈 수집, 스크립트 생성, 이미지 생성, 영상 번들 준비 등 장기 작업은 큐에서 처리합니다.
- 작업 상태는 `pending / running / success / failed`로 추적하며 재시도 가능한 구조를 전제로 합니다.

## 4. 프런트엔드 UX 방향

- 한국어 전용 사이드바 메뉴 구조
- 각 단계별 수동 추가 지시 입력 패널
- 근거 패널, 작업 로그 테이블, 상태 배지, 편집 가능한 텍스트 영역
- 데스크톱 우선이되 모바일에서는 상단 스크롤 내비게이션을 제공

## 5. 추후 확장 권장사항

- SQLAlchemy 모델에 맞춘 Alembic 마이그레이션 추가
- 실제 제공자별 인증/재시도/쿼터 관리 미들웨어 추가
- 스냅샷 캡처용 헤드리스 브라우저 워커 분리
- 프로젝트/스크립트 버전 비교용 diff 저장 최적화
- 인증/권한 체계와 감사 로그 추가
